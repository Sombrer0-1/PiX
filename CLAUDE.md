# Git 提交规范

## 核心原则：只提交纯净源码

提交代码时必须确保：
- **只提交源码文件**（.ts, .vue, .json, .md 等）
- **不提交依赖包**（node_modules/）
- **不提交构建产物**（dist/, release/, release-bugfix/, *.exe, *.blockmap 等）
- **不提交编译中间产物**（*.tsbuildinfo, *.log 等）

目标：别人 clone 仓库后执行 `npm install` → `npm run build` → 打包，即可生成相同的可执行文件。

检查清单：
1. 运行 `git status` 确认没有意外文件
2. 使用 `git add <具体文件路径>` 暂存，绝不使用 `git add .` 或 `git add -A`
3. 确认 .gitignore 已覆盖所有产物目录

# 开发规范

## 对话风格

- 保持回答简短精炼
- 提交、问题、PR 评论或代码中不使用表情符号
- 不使用浮夸或愉快的填充文本（例如，用"Thanks @user"而不是"Thanks so much @user!"）
- 仅使用技术性文字，直接表达
- 当用户提问时，先回答问题再进行编辑或运行实现命令
- 回应用户反馈或分析时，先明确表示同意或不同意，再说做了什么修改

## 代码质量

- 在进行大范围修改之前、编辑未完全检查的文件之前、以及被要求调查或审计时，必须完整阅读文件。不要依赖搜索片段进行大范围修改
- 除非绝对必要，否则不使用 `any` 类型
- 内联只有一个调用点的单行辅助函数
- 检查 node_modules 中的外部 API 类型，不要猜测
- **禁止内联导入**（`await import()`、`import("pkg").Type`、动态类型导入）。仅使用顶层导入
- 永远不要为了修复过时依赖的类型错误而移除或降级代码；而是升级依赖
- 在删除看起来是有意为之的功能或代码之前，务必询问
- 除非用户要求，否则不要保持向后兼容性

## 前端开发

- **Vuetify 组件**：使用 Vuetify 组件库设计前端 UI 时，必须先查看 `G:\develop\pi\vuetify_guide` 下的详细用法文档。该指南包含导航说明和正确用法模式，避免乱用出错。不要猜测组件 API 或属性 - 在实现之前先在指南中验证

## 命令

- 除非用户要求，否则不要运行 `npm run build` 或 `npm test`
- 如果创建或修改了测试文件，运行它并迭代测试或实现直到通过
- 对于临时脚本，将其写入临时文件（例如 `/tmp`），运行，需要时编辑，完成后删除。不要在 `bash` 命令中嵌入多行脚本
- 除非用户要求，否则不要提交

## 依赖和安装安全

- 将 npm 依赖和 lockfile 变更视为已审查的代码。直接外部依赖保持固定版本
- 本地安装/更新使用 `npm install --ignore-scripts`；干净/CI 风格使用 `npm ci --ignore-scripts`。除非用户要求，否则不要运行生命周期脚本
- 如果依赖元数据变更，使用 `npm install --package-lock-only --ignore-scripts` 刷新 `package-lock.json`

## Git

多个 pi 会话可能同时在此工作目录中运行，各自修改不同文件。触及你自身变更范围之外的未暂存、已暂存或未跟踪文件的 Git 操作会破坏其他会话的工作。遵循以下规则：

提交：

- 只提交你在本次会话中修改的文件
- 暂存明确的路径（`git add <path1> <path2>`）；永远不要使用 `git add -A` / `git add .`
- 提交前，运行 `git status` 验证你只暂存了自己的文件

永远不要运行（会破坏其他代理的工作或绕过检查）：

- `git reset --hard`、`git checkout .`、`git clean -fd`、`git stash`、`git add -A`、`git add .`、`git commit --no-verify`

如果发生变基冲突：

- 只解决你修改的文件中的冲突
- 如果冲突在你未修改的文件中，中止并询问用户
- 永远不要强制推送

## Issues 和 PRs

参见 `CONTRIBUTING.md` 了解贡献者门槛（自动关闭工作流、`lgtm`/`lgtmi`、质量标准）

审查 PRs 时：

- 除非用户明确要求，否则不要运行 `gh pr checkout`、`git switch` 或以其他方式将工作树切换到 PR 分支
- 使用 `gh pr view`、`gh pr diff`、`gh api` 和本地 `git show`/`git diff` 对已获取的引用来检查 PR 元数据、提交和补丁，而不切换分支
- 如果需要 PR 文件内容，将它们获取/读取到临时文件中，或使用 `git show <ref>:<path>` 而不切换分支

发布 issue/PR 评论时：

- 将评论写入临时文件，使用 `gh issue/pr comment --body-file` 发布（永远不要通过 `--body` 使用多行 markdown）
- 保持评论简洁、技术性，使用用户的语气
- 每条 AI 发布的评论都以发起提示指定的 AI 生成免责声明行结尾（例如 `This comment is AI-generated by `/wr``）

通过提交关闭 issues 时：

- 在消息中包含 `fixes #<number>` 或 `closes #<number>`，这样合并会自动关闭 issue。对于多个 issues，对每个 issue 重复关键字（`closes #1, closes #2`）；共享关键字（`closes #1, #2`）只关闭第一个
